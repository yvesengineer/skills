Case de negócio

Plataforma de Reservas de Experiências

Construa uma API backend para clientes navegarem por experiências com vagas limitadas, criarem reservas, consultarem reservas, cancelarem quando permitido e simularem pagamento.

Decisões técnicas, domínio e priorização
Contexto

Você foi chamado para construir o backend de uma plataforma de reservas.

A plataforma permite que clientes reservem experiências com vagas limitadas, como aula de surf, passeio de barco, workshop de culinária, tour guiado, day use em hotel e evento presencial.

Experiência Nome, descrição, preço, data e horário.
Disponibilidade Quantidade de vagas disponíveis por experiência.
Reserva Relação entre cliente, experiência e estado da solicitação.
Pagamento Simulação de uma etapa financeira para concluir ou recusar uma reserva.
Objetivo

Construa a melhor solução possível dentro do tempo disponível.

Você pode usar qualquer linguagem, framework, banco de dados, biblioteca, documentação, ferramenta de AI, IDE agentica ou recurso que preferir. O objetivo não é apenas terminar tudo, mas demonstrar como você pensa, prioriza, usa ferramentas e explica decisões.

Prefira entregar uma solução menor, funcional e bem pensada do que tentar implementar tudo de forma superficial.
Funcionalidades esperadas

A solução deve permitir, no mínimo, o fluxo principal de reserva.

  1. Listar experiências disponíveis.
  2. Consultar detalhes de uma experiência.
  3. Criar uma reserva para uma experiência.
  4. Consultar uma reserva.
  5. Cancelar uma reserva.
  6. Simular o pagamento de uma reserva.
Regras de negócio

O domínio tem decisões importantes, mesmo sendo um exercício curto.

Vagas limitadas

Uma experiência possui quantidade limitada de vagas. Não deve ser possível reservar uma experiência sem disponibilidade.

Associação obrigatória

Uma reserva deve estar associada a um cliente e a uma experiência.

Estados da reserva

Uma reserva pode ter diferentes estados, de acordo com as decisões que você tomar para o domínio.

Pagamento simulado

O pagamento pode ser simulado da forma que você considerar adequada.

Cancelamento

O cancelamento só deve ser permitido quando fizer sentido para as regras que você definir.

Premissas adicionais

Você pode assumir regras adicionais, desde que consiga explicá-las depois.

Liberdade técnica

As escolhas de solução fazem parte da avaliação.

  • Quais endpoints criar.
  • Como modelar os dados.
  • Como organizar o código.
  • Como armazenar as informações.
  • Como representar estados da reserva.
  • Como simular pagamento.
  • Quais validações implementar.
  • Quais testes escrever e limitações assumir.
O que será avaliado

Será avaliado tanto o resultado quanto o processo.

A avaliação considera como você entende o problema, prioriza, modela o domínio, desenha a API, organiza o código, lida com persistência, validações, erros, consistência, testes, comunicação e uso de AI.

Entendimento do problema Priorização Modelagem de domínio Contrato HTTP Consistência de dados Concorrência Pagamento Cancelamento Validações Testes Comunicação Uso de AI

Ao final, você deverá apresentar sua solução explicando o que foi implementado, como modelou o domínio, quais decisões técnicas tomou, quais regras assumiu, o que ficou de fora e o que faria diferente com mais tempo.

Tempo sugerido

Você terá entre 60 e 90 minutos.

1 Entenda e recorte

Faça perguntas, assuma premissas e defina um MVP viável.

2 Implemente o fluxo central

Liste experiências, crie reserva, consulte, cancele e simule pagamento.

3 Prove e explique

Rode a solução, mostre testes ou evidências e explique trade-offs.
